## 3 Функциональное проектирование

В данном разделе рассмотрено функционирование серверной части приложения. Проведен анализ входных и выходных данных, основных модулей программы и их зависимостей. Также описаны все классы, которые необходимо реализовать, рассмотрены их поля и методы.

Серверная часть приложения построена на основе слоистой архитектуры с использованием фреймворка FastAPI [17] и включает следующие уровни: модели данных (ORM-сущности SQLAlchemy), схемы валидации (Pydantic-модели для API [18]), репозитории (доступ к базе данных), сервисы (бизнес-логика) и представления (HTTP-эндпоинты). Внедрение зависимостей реализовано с помощью библиотеки Dishka [19].

Разрабатываемое приложение можно разделить на следующие модули:

- модуль аутентификации и авторизации;
- модуль управления компаниями и профилями;
- модуль парсинга данных;
- модуль ИИ-анализа;
- модуль поиска и истории;
- модуль списков отслеживаний и уведомлений;
- модуль сравнения компаний;
- модуль ИИ-скоринга;
- модуль экспорта данных;
- модуль администрирования;
- модуль конфигурации;
- модуль событий реального времени.

Диаграмма всех классов приложения представлена на чертеже ГУИР.400201.040 РР.1.

### 3.2 Базовые классы моделей данных

В качестве ORM-фреймворка используется SQLAlchemy версии 2.0 [20] с асинхронным движком asyncpg для работы с PostgreSQL [21]. Все модели данных образуют иерархию наследования, где базовые классы предоставляют общую функциональность: идентификацию, временные метки и хранение метаданных. Для аннотации типов используется механизм Mapped из SQLAlchemy 2.0.

**3.2.1** Класс `Model` является базовым классом для всех ORM-моделей приложения. Он наследуется от класса `SQLQuery` библиотеки advanced-alchemy и предоставляет общую функциональность для работы с базой данных PostgreSQL. Данный класс не содержит собственных полей и служит единой точкой расширения для всех моделей приложения.

**3.2.2** Класс `TimestampedModel` наследуется от класса `Model` и предназначен для автоматического отслеживания времени создания и последнего изменения записи в базе данных. Он содержит следующие поля:

- `created` типа `Mapped[datetime]` является публичным полем, содержит дату и время создания записи в формате `TIMESTAMP` с учетом часового пояса, значение задается автоматически при вставке записи в базу данных;
- `updated` типа `Mapped[datetime]` является публичным полем, содержит дату и время последнего изменения записи в формате `TIMESTAMP` с учетом часового пояса, значение обновляется автоматически при каждом изменении записи.

**3.2.3** Класс `IDModel` наследуется от класса `TimestampedModel` и добавляет уникальный идентификатор записи. Он содержит поле `id` типа `Mapped[str]`, является первичным ключом таблицы, генерируется автоматически на основе алгоритма ULID (Universally Unique Lexicographically Sortable Identifier – универсальный уникальный лексикографически сортируемый идентификатор) [22], что обеспечивает уникальность, сортируемость по времени и читаемость идентификаторов.

Класс `IDModel` реализует следующие публичные методы:

- `__eq__(self, other: object) -> bool` – сравнение двух объектов по значению поля `id`, возвращает `True`, если идентификаторы совпадают;
- `__hash__(self) -> int` – вычисление хеша объекта на основе значения поля `id` для использования в хеш-таблицах и множествах;
- `__repr__(self) -> str` – строковое представление объекта в формате `"ClassName(id=значение)"` для отладки и логирования.

**3.2.4** Класс `MetadataMixin` является примесью (от англ. mixin) и предназначен для добавления возможности хранения произвольных метаданных в любой модели. Он содержит публичное поле `meta` типа `Mapped[dict]`, которое хранит произвольные пары «ключ-значение» в формате `JSONB`, по умолчанию содержит пустой словарь. Для отслеживания изменений в словаре используется обертка `MutableDict` из SQLAlchemy.

Класс `MetadataMixin` реализует публичный метод `set_json_key(self, key: str, value: Any) -> None`, который совершает атомарное обновление отдельного ключа в `JSONB`-поле `meta` без перезаписи остальных ключей, помечает поле как измененное для корректного сохранения в базу данных.

**3.2.5** Класс `RecordModel` наследуется одновременно от двух классов: `IDModel` и `MetadataMixin`, объединяя идентификатор, временные метки и метаданные в единый базовый класс. Большинство моделей приложения наследуются именно от данного класса. Новых полей и методов в данном классе не объявляется.

### 3.3 Классы модуля аутентификации и авторизации

Модуль аутентификации и авторизации обеспечивает регистрацию, вход, управление токенами и проверку прав доступа пользователей. Поддерживается аутентификация по электронной почте с паролем и через Google OAuth [23]. Для хеширования паролей используется алгоритм bcrypt через библиотеку pwdlib. Токены представляют собой строки, хеш которых хранится в базе данных.

**3.3.1** Класс `User` наследуется от класса `RecordModel` и представляет учетную запись пользователя в системе. Он содержит следующие поля:

- `email` типа `Mapped[str]` является публичным полем, содержит адрес электронной почты пользователя, должен быть уникальным в пределах таблицы, используется для аутентификации;
- `hashed_password` типа `Mapped[str | None]` является публичным полем, содержит хеш пароля пользователя, вычисленный алгоритмом bcrypt, равен `None` для пользователей, зарегистрированных через OAuth-провайдер;
- `name` типа `Mapped[str]` является публичным полем, содержит отображаемое имя пользователя;
- `avatar_url` типа `Mapped[str | None]` является публичным полем, содержит URL аватара пользователя, может быть получен от OAuth-провайдера или загружен вручную;
- `is_superuser` типа `Mapped[bool]` является публичным полем, определяет, обладает ли пользователь правами администратора, по умолчанию равно `False`;
- `is_verified` типа `Mapped[bool]` является публичным полем, указывает, подтвержден ли адрес электронной почты пользователя, по умолчанию равно `False`;
- `is_enabled` типа `Mapped[bool]` является публичным полем, определяет, активна ли учетная запись, заблокированные пользователи не могут входить в систему, по умолчанию равно `True`;
- `oauth_provider` типа `Mapped[str | None]` является публичным полем, содержит название OAuth-провайдера, через который зарегистрирован пользователь (например, `"google"`), равно `None` для обычной регистрации по электронной почте;
- `oauth_id` типа `Mapped[str | None]` является публичным полем, содержит идентификатор пользователя у OAuth-провайдера, используется для связывания учетных записей;
- `search_quota` типа `Mapped[int]` является публичным полем, определяет максимальное количество запросов на профилирование компаний в сутки, по умолчанию равно 10, может быть изменено администратором;
- `notification_settings` типа `Mapped[NotificationSettings]` является публичным полем, содержит настройки уведомлений пользователя, хранится как `JSONB` с валидацией через Pydantic-модель `NotificationSettings`;
- `tokens` является отношением к модели `Token` (один ко многим), обеспечивает доступ ко всем токенам аутентификации пользователя;
- `search_history` является отношением к модели `SearchHistory` (один ко многим), обеспечивает доступ к истории поисковых запросов пользователя;
- `watchlists` является отношением к модели `Watchlist` (один ко многим), обеспечивает доступ ко всем спискам наблюдения пользователя.

**3.3.2** Класс `Token` наследуется от класса `RecordModel` и представляет токен аутентификации, привязанный к конкретному пользователю. Каждый токен имеет набор разрешений (от англ. scopes), определяющих доступные операции. Он содержит следующие поля:

- `user_id` типа `Mapped[str]` является публичным полем, содержит внешний ключ на таблицу пользователей, указывает, какому пользователю принадлежит данный токен;
- `token_hash` типа `Mapped[str]` является публичным полем, содержит SHA-256 хеш значения токена, сам токен не хранится в базе данных в открытом виде;
- `name` типа `Mapped[str]` является публичным полем, содержит отображаемое название токена, задаваемое пользователем при создании (например, `"Мой рабочий токен"`);
- `permissions` типа `Mapped[list[str]]` является публичным полем, содержит список разрешений, определяющих, какие операции доступны с данным токеном, хранится как `JSONB`-массив;
- `expires_at` типа `Mapped[datetime | None]` является публичным полем, содержит дату и время истечения срока действия токена, если равно `None`, то токен является бессрочным;
- `last_used_at` типа `Mapped[datetime | None]` является публичным полем, содержит дату и время последнего использования токена для аутентификации;
- `user` является отношением к модели `User` (многие к одному), обеспечивает доступ к пользователю-владельцу токена.

**3.3.3** Класс `AuthScopes` наследуется от класса `StrEnum` и определяет все области доступа системы авторизации. Данный класс не содержит полей экземпляра и имеет только строковые константы класса:

- `PROFILE_VIEW = "profile_view"` – просмотр профилей компаний;
- `PROFILE_CREATE = "profile_create"` – инициирование профилирования компаний;
- `PROFILE_EXPORT = "profile_export"` – экспорт профилей в формат PDF;
- `WATCHLIST_MANAGE = "watchlist_manage"` – управление списками отслеживания;
- `COMPARE = "compare"` – сравнение компаний;
- `ADMIN_USERS = "admin_users"` – управление пользователями в админ-панели;
- `ADMIN_SOURCES = "admin_sources"` – управление источниками данных в админ-панели;
- `ADMIN_METRICS = "admin_metrics"` – просмотр метрик использования системы;
- `FULL_CONTROL = "full_control"` – полный доступ ко всем функциям системы.

**3.3.4** Класс `AuthDependency` наследуется от класса `OAuth2PasswordBearer` библиотеки FastAPI и реализует зависимость (от англ. dependency) для извлечения и проверки токена аутентификации из HTTP-запроса. Он содержит следующие поля:

- `token_required` типа `bool` является публичным полем, определяет, является ли наличие токена обязательным, если равно `True`, то при отсутствии токена возвращается ошибка 401, по умолчанию равно `True`;
- `return_token` типа `bool` является публичным полем, определяет, возвращать ли объект токена вместе с объектом пользователя, по умолчанию равно `False`.

Класс `AuthDependency` реализует следующие публичные методы:

- `parse_token(self, header_token: str | None, provided_token: str | None) -> str | None` – извлечение значения токена из заголовка Authorization HTTP-запроса или из переданного параметра, возвращает строковое значение токена или `None`;
- `__call__(self, request: Request, security_scopes: SecurityScopes) -> tuple[User, Token] | User | None` – основная точка входа, вызывается фреймворком FastAPI при разрешении зависимости, извлекает токен из запроса, передает его в `AuthService` для проверки прав доступа и возвращает объект пользователя.

**3.3.5** Класс `PasswordHasher` реализует протокол `PasswordHasherProtocol` и предоставляет функции хеширования и проверки паролей с использованием алгоритма bcrypt. Он содержит приватное поле `_hasher` типа `PasswordHash`, которое является экземпляром хешера из библиотеки pwdlib, сконфигурированным с использованием `BcryptHasher` с 12 раундами хеширования.

Класс `PasswordHasher` реализует следующие публичные методы:

- `verify_password(self, plain_password: str, hashed_password: str) -> bool` – проверка соответствия открытого пароля его хешу, возвращает `True`, если пароль верный;
- `get_password_hash(self, password: str) -> str` – вычисление хеша для заданного пароля, возвращает строку с bcrypt-хешем.

**3.3.6** Класс `AuthService` отвечает за бизнес-логику аутентификации и авторизации пользователей. Он содержит следующие поля, внедряемые через конструктор посредством Dishka:

- `user_repo` типа `UserRepository` является публичным полем, содержит репозиторий для доступа к данным пользователей;
- `token_repo` типа `TokenRepository` является публичным полем, содержит репозиторий для доступа к данным токенов;
- `password_hasher` типа `PasswordHasher` является публичным полем, содержит сервис хеширования паролей.

Класс `AuthService` реализует следующие публичные методы:

- `async authenticate_user(self, email: str, password: str) -> tuple[User, int]` – проверка учетных данных пользователя при входе по электронной почте и паролю, возвращает кортеж из объекта пользователя и HTTP-кода статуса (200 при успехе, 401 при неверных данных);
- `async authenticate_oauth(self, provider: str, oauth_token: str) -> User` – аутентификация пользователя через OAuth-провайдер (Google), обменивает OAuth-токен на данные пользователя, если пользователь с данным oauth_id не существует, создает новую учетную запись;
- `async find_by_token(self, token_hash: str) -> tuple[User, Token] | None` – поиск пользователя по хешу токена аутентификации, возвращает кортеж из пользователя и токена или `None`, если токен не найден;
- `async find_user_and_check_permissions(self, header_token: str | None, security_scopes: SecurityScopes) -> tuple[User, Token]`&nbsp;– основной метод проверки аутентификации и авторизации, валидирует токен, проверяет, что запрашиваемые `security_scopes` входят в разрешения токена, проверяет активность учетной записи, при ошибке возбуждает исключение `HTTPException` с кодом 401 или 403;
- `async create_token(self, user: User, permissions: list[str], name: str, expires_at: datetime | None) -> str` – создание нового токена аутентификации для пользователя с указанными разрешениями и сроком действия, генерирует случайное значение токена, сохраняет его SHA-256 хеш в базу данных, возвращает открытое значение токена;
- `async revoke_token(self, token_id: str, user: User) -> None`&nbsp;– отзыв (удаление) токена аутентификации, проверяет, что токен принадлежит указанному пользователю;
- `async request_password_reset(self, email: str) -> str` – генерация токена для сброса пароля, сохраняет хеш токена в базу данных и возвращает открытое значение токена для отправки на электронную почту пользователя;
- `async reset_password(self, reset_token: str, new_password: str) -> None` – установка нового пароля по токену сброса, проверяет валидность токена и обновляет хеш пароля в записи пользователя;
- `async request_email_verification(self, user: User) -> str` – генерация шестизначного кода подтверждения электронной почты в формате `"XXX-XXX"`;
- `async verify_email(self, user: User, code: str) -> bool` – проверка кода подтверждения электронной почты, при успехе устанавливает поле `is_verified` пользователя в `True`.
